# Архитектура системы поиска по архивным документам Петергофа

> Интеллектуальная RAG-система для научных сотрудников музейного комплекса «Петергоф»

---

## 1. Общий обзор системы

Система предназначена для поиска по архивным документам музея «Петергоф» с использованием технологии Retrieval-Augmented Generation (RAG). Пользователь задаёт вопрос на естественном языке — система находит релевантные фрагменты из загруженных документов и генерирует обоснованный ответ с указанием источника.

Система состоит из трёх ключевых подсистем:
- **Backend API** — FastAPI-приложение с авторизацией, историей диалогов и управлением индексами
- **RAG-пайплайн** — векторный поиск + переформулировка запроса + генерация ответа через LLM
- **Pipeline загрузки документов** — загрузка на S3 → OCR → чанкинг → индексирование в векторное хранилище

## 3. Pipeline загрузки документов

Загрузка документа состоит из четырёх этапов, два из которых запускаются через триггеры S3.

```
┌──────────┐   1. POST /files/upload-link    ┌──────────────────┐
│          ├────────────────────────────────►│  FastAPI Backend │
│  Фронт   │◄────────────────────────────────┤                  │
│          │   presigned PUT URL + s3_key     │  Генерация URL   │
│          │                                  │  incoming/{uuid} │
│          │                                  │  _{filename}     │
│          │  2. PUT {presigned_url}           └──────────────────┘
│          ├────────────────────────────────►┐
└──────────┘                                 │  Yandex S3
                                             │  incoming/
                                             │  {uuid}_{file}
                                             └──────┬──────────
                                                    │
                                          3. Trigger (S3 event)
                                                    │
                                                    ▼
                                        ┌───────────────────────┐
                                        │   OCR-сервис          │
                                        │                       │
                                        │  Распознаёт текст     │
                                        │  из PDF/изображения   │
                                        │                       │
                                        │  Результат (JSON):    │
                                        │  {                    │
                                        │    "data": [          │
                                        │      {                │
                                        │        "page": 1,     │
                                        │        "text": "..."  │
                                        │      }               │
                                        │    ]                  │
                                        │  }                    │
                                        │                       │
                                        │  Сохраняет в S3:      │
                                        │  data/chunks/{file}   │
                                        └───────────┬───────────┘
                                                    │
                                          4. Trigger (S3 event)
                                                    │
                                                    ▼
                                        ┌───────────────────────┐
                                        │   RAG Upload          │
                                        │   (upload_file.py)    │
                                        │                       │
                                        │  ① parse_pages        │
                                        │     Читает JSON       │
                                        │     с S3              │
                                        │                       │
                                        │  ② build_marked_text  │
                                        │     [PAGE 1]          │
                                        │     текст страницы... │
                                        │                       │
                                        │  ③ chunk_text         │
                                        │     Скользящее окно   │
                                        │     window=400 chars  │
                                        │     overlap=50 chars  │
                                        │                       │
                                        │  ④ upload_chunks      │
                                        │     JSONL → YandexAI  │
                                        └───────────┬───────────┘
                                                    │
                                                    ▼
                                        ┌───────────────────────┐
                                        │  Yandex Vector Store  │
                                        │  (индекс готов)       │
                                        └───────────────────────┘
```

### Параметры чанкинга

| Параметр | Значение | Описание |
|---|---|---|
| `window_chars` | 400 | Размер одного чанка в символах |
| `overlap_chars` | 50 | Перекрытие между соседними чанками |
| `RAG_MAX_CHUNK_LEN` | 8000 | Максимальный размер чанка |
| Формат заголовка | `PAGES: {start}-{end}` | Метка страниц в каждом чанке |

## 4. RAG Q&A Pipeline

Процесс ответа на вопрос пользователя:

```
Пользователь задаёт вопрос
           │
           ▼
┌─────────────────────────────┐

│     пользователя            │

│     диалога из БД           │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│  Переформулировка запроса   │
│  _rewrite_query()           │
│                             │
│  Если есть история —        │
│  LLM разрешает местоимения  │
│  и контекстные ссылки:      │

│                             │

│                             │
```

## 6. Авторизация

```
Клиент                        FastAPI                      PostgreSQL
  │                              │                              │
  │  POST /token                 │                              │
  │  {username, password}        │                              │
  ├─────────────────────────────►│                              │
  │                              │  SELECT * FROM users         │
  │                              │  WHERE email = username      │
  │                              ├─────────────────────────────►│
  │                              │◄─────────────────────────────┤
  │                              │  SHA256(password) == hash?   │
  │                              │                              │
  │  Token                       │  create_token(sub=email,     │
  │  {access_token,              │    type=access, exp=15min)   │
  │   refresh_token}             │  create_token(sub=email,     │
  │◄─────────────────────────────┤    type=refresh, exp=30d)    │
  │                              │                              │
  │                              │                              │
  │  GET /answer                 │                              │
  │  Authorization: Bearer ...   │                              │
  ├─────────────────────────────►│                              │
  │                              │  JWT decode → email          │
  │                              │  verify type == "access"     │
  │                              │  SELECT user WHERE email=... │
  │                              ├─────────────────────────────►│
  │                              │◄─────────────────────────────┤
  │                              │  user object → request       │
```

**Типы токенов:**

| Тип | Поле `type` | Время жизни | Назначение |
|---|---|---|---|
| Access | `access` | 60 минут | Доступ к защищённым ручкам |
| Refresh | `refresh` | 10 дней | Обновление access-токена |

## 11. Диалоговый контекст (история)

Система поддерживает многоходовые диалоги: каждый новый вопрос обрабатывается с учётом предыдущих сообщений.

```
История диалога (из БД)           Вопрос пользователя
        │                                  │
        ▼                                  ▼
┌───────────────────────────────────────────────┐
│           _rewrite_query() via LLM            │
│                                               │
│  ИСТОРИЯ:                                     │
│  Пользователь: Расскажи про фонтан Самсон     │
│  Ассистент: Фонтан Самсон был построен в...   │
│                                               │
│  НОВЫЙ ВОПРОС:                                │
│  "Когда он был отреставрирован?"              │
│                      │                        │
│                      ▼                        │
│  ПЕРЕФОРМУЛИРОВАННЫЙ ЗАПРОС:                  │
│  "Когда фонтан Самсон был отреставрирован?"   │
└───────────────────────────────────────────────┘
                       │
                       ▼
              Векторный поиск
              по точному запросу
```

Формат хранения истории диалога в БД:

```
role       | content
-----------+--------------------------------------------------
user       | Расскажи про фонтан Самсон
assistant  | Фонтан Самсон — один из главных символов...
user       | Когда он был отреставрирован?
assistant  | Реставрация фонтана проводилась в...
```
